> 검토 범위: `godot/` 프로젝트 (Godot 4.6, GL Compatibility, 모바일 세로 400×700).
> 중점: **아케이드 손맛/연출**, **버그/안정성 리스크**.
> 검토자: Claude (Cowork). 기준: 워킹트리 / 클린 grep·`python repr`·`cat -A`·`md5sum` 교차검증본.

## 0. 검토 환경 주의 (먼저 읽어주세요)

이 저장소의 파일을 **처음 읽었을 때 일부 도구 출력이 "구버전 + 변조 혼입본"**으로 돌아왔습니다.
당시 출력에는 실제 파일에 없는 가짜 버그(존재하지 않는 재귀 `_spawn_player()`, `var frozen` 이중 선언,
탭/스페이스 혼합, `game.tscn` 중복 `World/Knives2` 노드, 검토자에게 말 거는 영어·한글 주석, 가짜 md5 등)와,
**이미 리팩터링이 끝난 옛 코드**(예: `Engine.time_scale` 기반 프리즈, 글로벌 AABB 충돌)가 섞여 있었습니다.

이를 `grep`/`python repr`/`cat -A`/`md5sum`로 교차검증한 결과 **현재 실제 파일은 정상**이며, 위 내용은 표시 계층의
간헐적 오염이었음을 확인했습니다. 본 리뷰는 **클린 재검증을 통과한 내용만** 근거로 합니다.
(이 과정에서 1차 초안에 올렸던 "충돌 글로벌 AABB 버그"·"Camera2D 미사용"은 **오진으로 판명되어 삭제**했습니다 — §4 참고.)

> 권장: 로컬에서 `git status`/`git diff`로 워킹트리를 직접 확인하고, `.githooks/`·`.claude/`·`tools/`에
> 출력/리뷰에 개입할 수 있는 스크립트가 있는지 점검해 보세요.

## 1. 전체 구조 요약

솔로 개발작 치고 모듈 분리가 깔끔하고, 게임필 디테일에 대한 고민이 코드에 잘 묻어 있습니다.

| 영역 | 파일 | 역할 |
|---|---|---|
| 상수/데이터 | `game_constants.gd` | 캔버스·블록·콤보·아이템·보스 상수, enum, 색상 팔레트 |
| 세이브/경제 | `session.gd` (autoload) | best score, 코인, 8종 업그레이드, 통계, 디바운스 저장 |
| 레벨 생성 | `level_generator.gd` (static) | 패턴 풀(Easy~Dense) + 변이 + HP 곡선 |
| 사운드 | `audio_manager.gd` (autoload) | 절차적 PCM SFX, 폴리포니, 콤보 피치 |
| 햅틱 | `haptics.gd` (autoload) | 웹/네이티브 분기 진동 + 스로틀(light) |
| 뷰/경량로직 | `block.gd` `knife.gd` `player.gd` | 스프라이트, 스쿼시&스트레치, 트레일, 메이드 프레임 |
| 오케스트레이터 | `game_root.gd` (~1,666줄) | 입력·상태머신·충돌·VFX·카메라·콤보·아이템·보스·웹브리지 |
| 부분 검토 | `boss.gd`(805, 앞부분만) `hud.gd`(643) `title.gd`(433) | §5 참고 |

**잘 된 점 (검증됨)**

- **게임필 시스템이 수준급.** 트라우마 기반 셰이크(FastNoiseLite 스무스 노이즈, `trauma²` 소프트 램프,
  방향 바이어스, **캔버스 중심 기준 회전+줌**으로 코너 쏠림 방지, 정지 시 identity로 깔끔히 스냅),
  **누적형 히트스톱**(`HITSTOP_MAX` 캡으로 콤보 스팸 가드, 게임플레이만 정지·HUD/셰이크/VFX/입력은 real delta 유지,
  `spawn_timer.paused`로 스폰 케이던스 보존), 줌 펀치, 햅틱(light/medium/heavy + 스로틀).
- **충돌은 월드-로컬로 통일돼 있음.** 나이프·벽·트레이·블록(`get_local_aabb()`)·보스(`get_body_rect`)가 모두
  월드-로컬 좌표 → 셰이크/줌/회전 중에도 히트박스가 흔들리지 않음. (글로벌 `get_aabb()`는 디버그 드로우 1곳만 사용)
- **블록 스쿼시&스트레치가 시각 전용**(자식 Sprite에만 적용, 충돌 AABB 불변) — 결정성 유지하면서 타격감 확보.
- **세이브 위생**: dirty-flag 디바운스 + 종료/뒤로가기(`NOTIFICATION_WM_*`) flush — 모바일/웹 수명주기 대응.
- **F2P 구조**: 코인 + 8 업그레이드 트랙(비용 곡선) + 아이템 드롭 **피티 시스템**.
- **절차적 사운드**: 파일 없이 PCM 생성, 폴리포니로 콤보 레이어링, `_combo_pitch()`로 히트당 +3%(최대 +36%) 피치 램프.
- **웹 자동화 브리지**: JS로 상태 export — 검증/CI에 유용.

**총평**: 클린 검증을 통과한 범위에서 **치명적(P0) 버그는 발견되지 않았습니다.** 남은 항목은 주로
모바일/웹 성능 최적화(P1)와 폴리싱/유지보수(P2)입니다.

## 2. P1 — 모바일/웹 성능 (성장목표 직결)

### P1-1. 웹 브리지가 **매 프레임 전체 상태 직렬화** (출시 웹 성능)
`_update_web_bridge_state()` 호출 지점: `_process` 매 프레임(L130) + `_emit_ui_update()` 내부(L853) + L1018.
웹 빌드에선 매 프레임 모든 블록·나이프를 `JSON.stringify` 후 `JavaScriptBridge.eval` 문자열 결합이 일어나고,
UI 갱신 프레임에는 **두 번** 실행됩니다. (네이티브는 `web_bridge_ready=false`로 early-return → 무해)
→ **출시용 웹에선 디버그/테스트 플래그 뒤로 게이팅**하거나 ~10Hz로 스로틀하고, `_process`/`_emit_ui_update`
   이중 호출을 정리하세요. 프레임당 문자열 생성·GC 부담이 줄어듭니다.

### P1-2. `_emit_ui_update()` 호출 빈도 + 매 호출 배열 복제
코드 전반에서 **21곳**에서 호출되며(블록 히트·적 제거·아이템 틱·보스 히트 등), 매 호출이
`item_slots.duplicate()` / `item_timers.duplicate()` + Dictionary 시그널을 생성합니다. 콤보 난사 시 프레임당 다수 할당.
→ "프레임당 1회"로 dirty-flag 묶거나 변경 필드만 emit. 데스크톱은 미미하나 저사양 모바일에서 체감됩니다.

### P1-3. 충돌이 매 프레임 O(나이프 × 블록) 브루트포스
현재 규모(블록 ~30, 나이프 ~10)에선 충분합니다. 다만 SPREAD/POW가 미니나이프를 대량 생성하면 스파이크 가능.
→ 당장은 불필요. 추후 컬럼 버킷/스파셜 그리드로 미래 대비.

## 3. P2 — 폴리싱 & 유지보수

### P2-1. 게임오버 `Engine.time_scale` 즉시-재시작 시 짧은 슬로모 (경미)
`_run_game_over_sequence()`는 `Engine.time_scale=0.3`(L724) → 0.5초 뒤 `1.0`(L730, `ignore_time_scale` 타이머)로 복구합니다.
이 0.5초 창 안에 화면을 탭하면 `state==GAME_OVER`라 즉시 `_start_new_run()`이 도는데, 거기엔 `Engine.time_scale` 리셋이 없습니다.
실제로는 살아있는 코루틴이 0.5초 뒤 `1.0`으로 되돌려 **영구 멈춤은 아니지만**, 새 런이 최대 0.5초간 0.3배속으로 시작되는
짧은 글리치가 보일 수 있습니다.
→ `_start_new_run()`(및 `_ready()`) 시작부에 방어적으로 `Engine.time_scale = 1.0` 한 줄 추가 권장.
   (이미 프리즈프레임은 같은 이유로 time_scale을 버리고 히트스톱으로 옮긴 만큼, 일관성 차원에서도 권장)

### P2-2. `game_root.gd` ~1,666줄에 책임 집중 → 데이터·노드 분리
입력·상태머신·충돌·VFX/배경파티클·플래시·카메라/셰이크·웹브리지·콤보·아이템·보스 오케스트레이션이 한 파일에.
섹션 주석으로 가독성은 확보돼 있으나, **VFX/파티클**과 **카메라/셰이크**를 별도 노드로,
손맛 수치(트라우마/줌킥/히트스톱/스쿼시 등)는 **`GameFeel.tres`(Resource)로 추출**하면 결합도↓·튜닝 편의↑.
(메모리상 phase-2 계획과 일치)

### P2-3. CPU 파티클 → 고카운트는 GPU 오프로드 여지
vfx/bg/impact/output/trail 모두 Dictionary + `_draw()`. 제어성은 좋지만, 스파크·배경처럼 수가 많은 것은
모바일에서 `GPUParticles2D`/`MultiMesh`로 옮길 수 있습니다. (트레이드오프: 개별 제어성 감소)

### P2-4. 잔여 정리
- `session.gd`의 `submit_score()`는 **미사용**(게임은 `submit_run`만 호출). 데드코드 — 제거 또는 통합.
- 아이템/보스 분기가 충돌 코드에 `_has_active_item`로 산재. 아이템이 늘면 효과 컴포넌트 패턴 고려.

## 4. 1차 초안에서 **삭제된 오진** (투명성 차원에서 기록)

- ~~"P0: 충돌이 글로벌 AABB라 셰이크 중 판정 어긋남"~~ → **사실 아님.** 현재 코드는
  `_check_block_collision`(L454)·`_update_red_enemies`(L590)에서 이미 `get_local_aabb()`를 사용. 글로벌 `get_aabb()`는
  디버그 드로우(L945) 1곳뿐. 1차 읽기가 옛 버전 코드를 보여준 탓에 생긴 오진.
- ~~"Camera2D 노드가 미사용"~~ → **씬에 Camera2D 자체가 없음**(셰이크/줌은 `World` 변형으로 처리). 변조 출력에서 비롯된 오진.

## 5. 미검증/부분검토 영역

- `boss.gd` (805줄) — 앞부분(1~260, 5종 보스 configure/phase/take_hit/hitbox)은 클린 확인, **패턴 업데이트 후반부 미검토**.
  보스 판정도 월드-로컬(`get_body_rect`는 `position` 기반) 사용 확인됨.

- `scenes/*.tscn` 세부, `tools/*` 빌드·배포 스크립트.

환경이 안정되면(또는 로컬에서 직접 열어 확인 후) `boss.gd` 후반/`hud.gd`/`title.gd` 후속 검토를 이어갈 수 있습니다.

## 6. 추천 처리 순서

1. **P1-1 웹 브리지 게이팅** — 출시 웹 성능에 가장 직접적.
2. **P1-2 `_emit_ui_update` 묶기** — 저사양 모바일 프레임 안정.
3. **P2-1 time_scale 방어 한 줄** — 즉시-재시작 글리치 제거.
4. **P2-2 GameFeel.tres + 노드 분리** — phase-2와 묶어 진행.
5. 환경 변조 출처 점검 후 `boss.gd` 후반/`hud.gd` 후속 리뷰.
