0큰 그림 · 준비물

Play 스토어는 웹 빌드(.zip)를 받지 않습니다. 오직 서명된 Android App Bundle(.aab)만 받습니다. 시작 전 아래 5가지가 손에 있어야 합니다.

준비물HiteZero 현재 상태위치 / 값
① 개발자 계정 ($25 1회)확인 가입·신원확인 완료 여부play.google.com/console
② 서명된 AAB로컬에 존재 · CI로도 생성 가능build/android/hitezero.aab
③ 그래픽 에셋아이콘·피처·스크린샷 4장 준비됨store_assets/
④ 개인정보처리방침 URL라이브https://hitezero.netlify.app/privacy
⑤ 등록정보 텍스트EN/KR 확정안 보유본 문서 3장

꼭 알아야 할 3가지 개념

  • AAB(App Bundle) — Play에 올리는 패키징 형식. APK가 아니라 .aab여야 합니다. 우리 Android 프리셋은 이미 AAB(export_format=1)로 설정돼 있습니다.
  • 업로드 키(upload key) — 내가 보관하는 키스토어로 AAB에 서명합니다. HiteZero는 hitezero_upload.jks(별칭 upload)를 사용하며, GitHub CI 시크릿에도 등록돼 있습니다.
  • Play 앱 서명(Play App Signing) — 첫 업로드 시 켜기. 구글이 진짜 배포 키를 보관하므로, 업로드 키를 잃어도 재설정으로 복구할 수 있습니다.
이 문서는 v1.0 = 광고 0 · 인앱결제 0 · 데이터 수집 0의 "깨끗한" 출시 프로필을 기준으로 합니다. 이 프로필은 가장 단순하고 정책상 방어가 쉽습니다. 나중에 광고로 수익화하려면 같은 docs/ 폴더의 google_admin_manual.html(AdMob·AdSense 가이드)을 따르고, 데이터 보안·광고 선언을 모두 갱신해야 합니다.

1서명된 AAB 확보 — 어떤 빌드를 올릴까

Play에 올릴 단 하나의 파일입니다. 패키지가 com.gghf.hitezero(adtest 아님)이고 릴리스 서명된 AAB여야 합니다.

방법 A — GitHub Actions CI (권장 · 서명 일관성)

  1. 워크플로 실행. 버전 태그를 푸시하면 자동 빌드됩니다: 
    git tag v1.0.0
git push origin v1.0.0
    또는 GitHub Actions 탭 → "Android (AAB + APK)" → Run workflow 수동 실행.
  2. 아티팩트 다운로드. 실행 완료 후 Artifacts → hitezero-android 에서 hitezero.aab 다운로드. 서명은 리포 시크릿(ANDROID_KEYSTORE_BASE64 / ANDROID_KEY_ALIAS=upload / ANDROID_KEYSTORE_PASSWORD)으로 자동 처리됩니다.

방법 B — 로컬 빌드 (본인 머신, Android SDK 필요)

cd godot
export GODOT_ANDROID_KEYSTORE_RELEASE_PATH="$HOME/keystores/hitezero_upload.jks"
export GODOT_ANDROID_KEYSTORE_RELEASE_USER="upload"
export GODOT_ANDROID_KEYSTORE_RELEASE_PASSWORD="••••••"   # 셸 히스토리 주의

godot --headless --path . --export-release "Android" ../build/android/hitezero.aab

사전 1회: Godot 에디터에서 Project → Install Android Build Template… + Editor Settings에 Android SDK / JDK 17 경로 지정.

ad-test 빌드를 올리지 마세요. 현재 작업 브랜치(neon-style-7cc31)에는 AdMob 애드온이 들어와 있습니다. 프리셋 "Android Test (Ads)"는 일부러 다른 패키지 com.gghf.hitezero.adtest + APK로 분리돼 있으니, Play 출시용으로는 반드시 "Android" 프리셋(com.gghf.hitezero, AAB)만 사용하세요. 가장 안전한 길은 방법 A(CI)로 깨끗하게 서명된 AAB를 받는 것입니다.
업로드 형식 확인. 받은 파일이 .apk가 아니라 .aab인지, 그리고 디버그 서명이 아닌 릴리스 서명인지 확인하세요. (에디터 수동 export 시 "Export With Debug" 체크는 반드시 해제.)

2개발자 계정 + 앱 만들기

  1. 개발자 계정 등록. play.google.com/console → 최초 1회 $25 결제 + 신원확인. (개인/사업자 선택. 신원확인 승인에 시간이 걸릴 수 있으니 가장 먼저 처리.)
  2. 앱 만들기. 콘솔 → 앱 만들기(Create app):
    • 앱 이름: HiteZero
    • 기본 언어: 영어(en-US) 확정 — 글로벌 노출 최대화. 출시 후 ko-KR 추가.
    • 앱 또는 게임: 게임(Game)
    • 무료 또는 유료: 무료(Free)
    • 선언 체크박스(개발자 프로그램 정책 / 미국 수출법) 동의
  3. 대시보드 따라가기. 앱 생성 후 콘솔이 "앱 설정 → 스토어 등록정보 → 출시" 순서의 체크리스트를 보여줍니다. 본 문서 3~8장이 그 순서를 그대로 따릅니다.
한국어 스토어 노출을 원하면, 출시 후 스토어 등록정보 → 언어 추가 → 한국어(ko-KR)로 3장의 KR 텍스트를 넣으면 됩니다. 영어 기본 + 한국어 추가가 노출/국내 커버 모두 잡는 조합입니다.

최종 라벨: "데이터가 수집되지 않음 · 데이터가 공유되지 않음 (No data collected / No data shared)".
근거: 네트워크 코드·INTERNET 권한·광고/분석 SDK 전무. 유일한 저장은 기기 로컬 user://save.cfg(최고점수·스테이지·코인·업그레이드·통계·음소거)로 외부 전송이 없어 Play 정의상 '수집'이 아닙니다. 개인정보처리방침(§1·§5)과 일치.
광고를 넣는 순간 이 답은 전부 바뀝니다. AdMob은 광고 ID(AAID)를 수집하므로 데이터 보안 = "예", 광고 = "예", AD_ID 권한 = "예"로 다시 신고해야 합니다. 그 경로는 google_admin_manual.html 5장 참고.

8출시 트랙 · Play 앱 서명

처음부터 프로덕션에 올리지 말고 내부 테스트 → (비공개 테스트) → 프로덕션 순으로 승격합니다.

  1. 내부 테스트 트랙 생성. 테스트 및 출시 → 테스트 → 내부 테스트 → 새 버전 만들기. 1장에서 받은 hitezero.aab 업로드.
  2. Play 앱 서명 켜기. 첫 업로드 시 자동 안내 → 켜기. 구글이 배포 키를 관리하고, 우리는 업로드 키 hitezero_upload.jks(별칭 upload)로만 업로드합니다.
  3. 업로드 인증서 확인. 업로드 후 Play Console에 등록된 업로드 인증서 SHA-256이 아래와 일치하는지 확인: 
    8F:21:9B:30:0C:B8:1F:74:06:31:A7:F2:60:7C:2D:6E:
81:84:7E:86:6A:71:27:C5:88:24:38:44:C8:42:A1:62
    첫 업로드가 업로드 키를 영구 고정하므로, CI 시크릿이 이 키스토어로 디코드되는지(keytool -list -v) 사전 확인 권장.
  4. 테스터 추가 + 실기기 검증. 본인 이메일을 테스터로 추가 → 받은 링크로 실기기 설치 → 1회 풀플레이(Boot→Title→Game, 보스·하트·점수 저장) 확인.
  5. 프로덕션 승격. 이상 없으면 비공개 테스트(선택) → 프로덕션으로 승격. 첫 제출은 구글 심사에 며칠 소요될 수 있습니다.
타깃 API 레벨. 현재 프리셋 target_sdk=35. Play의 신규 앱 타깃 API 요구는 매년 오르니, 등록 직전 Console이 요구하는 최신 레벨을 확인하고 필요 시 target_sdk를 올려 재빌드하세요.
키스토어 백업. hitezero_upload.jks와 비밀번호(~/keystores/hitezero_upload.password.txt)를 오프라인에 백업하세요. 분실 시 업로드 키 재설정은 가능하지만 번거롭습니다. git/dist에는 절대 커밋 금지.

9업데이트(2차 출시) 올리는 법

출시 후 새 버전을 올릴 때마다 단 하나만 기억하면 됩니다: versionCode를 +1.

  1. 버전 올리기. godot/export_presets.cfg 의 Android 프리셋: 
    version/code=2          ; ← 직전 업로드보다 반드시 +1
version/name="1.0.1"    ; 사람이 읽는 버전(자유)
    같은 version/code를 재업로드하면 Play가 거부합니다.
  2. AAB 재빌드 + 서명 (1장과 동일, CI면 새 v1.0.1 태그 푸시).
  3. 같은 트랙에 새 버전 업로드 → 출시 노트 작성 → 검토 → 출시.
스토어 텍스트·스크린샷만 바꾸는 변경은 AAB 재업로드 없이 스토어 등록정보 수정만으로 가능합니다(versionCode 무관).

10제출 전 최종 체크리스트

  • 개발자 계정($25) + 신원확인 완료
  • 업로드 파일이 서명된 .aab이고 패키지 com.gghf.hitezero (adtest 아님)
  • 앱 만들기: 이름 HiteZero · 기본 언어 en-US · 게임 · 무료
  • 스토어 등록정보: 제목·짧은/긴 설명(EN, +ko-KR) 입력
  • 그래픽: 아이콘 512 · 피처 1024×500 · 스크린샷 2장+ (권장 4장)
  • 개인정보처리방침 URL 등록 (https://hitezero.netlify.app/privacy)
  • 앱 액세스 = 모든 기능 제한 없이 제공
  • 광고 = 없음 / 인앱결제 = 없음
  • 콘텐츠 등급 설문 제출 (판타지/만화 폭력=예, 나머지=아니요)
  • 타깃 연령대 = 13+ (만 13세 미만 미포함)
  • 데이터 보안 = "데이터 수집 안 함 / 공유 안 함"
  • 정부/금융/건강/뉴스/COVID = 전부 아니요
  • 미국 수출법 준수 체크
  • Play 앱 서명 켜기 + 업로드 인증서 SHA-256 8F:21:…:A1:62 일치 확인
  • 내부 테스트로 실기기 1회 풀플레이 OK → 프로덕션 승격
  • 키스토어 + 비밀번호 오프라인 백업 완료

11자주 막히는 곳

증상원인 · 해결
"웹 zip 올렸는데 거부됨"당연 — Play는 .aab만 받음. 1장으로 AAB 빌드.
버전 코드 충돌같은 version/code 재업로드 불가. 항상 +1.
서명 안 된 빌드 거부에디터 export에서 "Export With Debug"가 켜져 있었음. 해제 후 릴리스로 재빌드(또는 CI 사용).
AAB 대신 APK가 나옴프리셋 export_format이 0으로 되돌아감. Android 프리셋은 1(AAB)이어야 함.
패키지가 .adtest로 올라감"Android Test (Ads)" 프리셋으로 빌드한 것. 출시는 "Android" 프리셋만 사용.
업로드 키 분실Play Console에서 업로드 키 재설정 요청 가능(앱 서명 키는 구글 보관 → 앱은 안 죽음). 그래도 .jks 백업.
타깃 API 경고Console이 요구하는 최신 레벨로 target_sdk 올려 재빌드.
데이터 보안 ↔ 권한 불일치 경고광고를 넣은 빌드인데 "수집 안 함"으로 신고하면 충돌. v1.0은 권한에 INTERNET/AD_ID가 없어야 정합.

12절대 하면 안 되는 것

ad-test 빌드(com.gghf.hitezero.adtest / APK)를 출시 트랙에 올리기 — 패키지가 다르고 광고 SDK가 섞임. 출시는 "Android" 프리셋(com.gghf.hitezero, AAB)만.
판타지/만화 폭력을 '아니요'로 낮춰 신고 — 과소신고는 등급 취소·앱 제거 사유. 칼 투척 = 만화적 폭력 '예'가 정답.
광고 넣고도 "데이터 미수집"으로 신고 — 정책 위반. 광고 도입 시 데이터 보안·광고·AD_ID를 모두 "예"로 갱신.
키스토어/비밀번호를 git·dist에 커밋 — 시크릿 유출. repo 바깥(~/keystores/)에 두고 오프라인 백업.
같은 versionCode 재업로드 — 무조건 거부. 업로드마다 +1.
