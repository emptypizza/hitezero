0현재 상태 한눈에

지금 만들어진 테스트 APK가 무엇이고, 무엇을 바꿔야 실제 수익이 나는지부터 정리합니다.

왜 테스트 ID부터? 테스트 광고는 항상 100% 노출되어 배치/레이아웃 검증에 안전합니다. 실제 광고 ID를 개발 중에 클릭·노출하면 계정 정지 위험이 있으므로, 실 ID 적용 후에는 반드시 테스트 기기 등록(3장)을 해야 합니다.

이 매뉴얼이 가정하는 것

  • 구글 계정(team.gghf@gmail.com 등)이 있고 결제수단을 등록할 수 있음
  • 출시용 업로드 키스토어는 이미 보유 (CI/문서에 기록됨 — ~/keystores/hitezero_upload.jks)
  • 웹 배포 도메인이 있음 (예: hitezero.netlify.app) — app-ads.txt/AdSense에 필요

1AdMob 콘솔 설정 (Android 앱)

apps.admob.com 에서 진행합니다. 모바일 앱(Android/iOS) 광고는 AdMob, 웹 광고는 AdSense로 서로 다른 제품입니다.

  1. AdMob 계정 생성/로그인. apps.admob.com 접속 → 구글 계정으로 로그인 → 국가·시간대·결제 통화 설정. (AdMob 가입 시 AdSense 계정이 함께 연결됩니다.)
  2. 결제 정보 등록. 좌측 지급(Payments) → 주소·세금정보·수령 계좌 입력. (지급 기준액 도달 전이라도 미리 등록 권장.)
  3. 앱 추가. 좌측 앱(Apps) → 앱 추가 → 플랫폼 Android 선택.
    • 이미 Google Play에 등록했다면 "예" → 패키지 com.gghf.hitezero 검색해 연결
    • 아직 미등록이면 "아니요" → 앱 이름 HiteZero 직접 입력
  4. 앱 ID(App ID) 확보. 앱 생성 직후 ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY 형식의 App ID가 발급됩니다. 이 값을 메모해 두세요. (구분자가 물결표 ~)
  5. 배너 광고 단위 생성. 해당 앱 → 광고 단위(Ad units) → 광고 단위 추가 → 형식 배너(Banner) 선택 → 이름 예: hitezero_bottom_banner → 생성. 발급되는 ca-app-pub-XXXX/ZZZZZZZZZZ 형식(구분자 슬래시 /)의 광고 단위 ID도 메모.
  6. (권장) 리워드 광고 단위도 생성. 이 프로젝트엔 "광고 보고 부활(rewarded revive)" 기능이 준비돼 있습니다. 형식 보상형(Rewarded)으로 하나 더 만들어 두면 나중에 바로 연결 가능.
이 단계가 끝나면 손에 App ID 1개 + 배너 광고 단위 ID 1개(+선택 리워드 ID)가 있어야 합니다. 다음 장에서 코드에 넣습니다.

2코드에 실 광고 ID 반영

테스트 ID를 내 ID로 교체하는 작업입니다. 우리 프로젝트에서 정확히 두 곳만 바꾸면 됩니다.

① App ID — android_export.cfg

파일: godot/addons/AdmobPlugin/android_export.cfg

[General]
is_real=true                  ; ← false에서 true로 변경 (실 ID 사용)

[Debug]
app_id="ca-app-pub-3940256099942544~3347511713"   ; 테스트용은 그대로 둬도 됨

[Release]
app_id="ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY"    ; ← 내 AdMob App ID

is_real=true면 빌드 시 [Release]의 App ID가 AndroidManifest에 주입됩니다.

② 배너 광고 단위 ID — banner_ads.gd

파일: godot/scripts/banner_ads.gd — _ready() 안 배너 설정 부분

_admob = Admob.new()
_admob.banner_position = LoadAdRequest.AdPosition.BOTTOM
_admob.banner_size = LoadAdRequest.RequestedAdSize.BANNER
_admob.banner_anchor_to_safe_area = true
_admob.android_real_banner_id = "ca-app-pub-XXXX/ZZZZ"   # ← 추가: 내 배너 단위 ID
# (android_debug_banner_id 는 테스트 ID 기본값이라 그대로 둠)
디버그 vs 릴리스 자동 전환. 플러그인은 디버그 빌드면 android_debug_banner_id(테스트), 릴리스 빌드면 android_real_banner_id(실제)를 사용합니다. 따라서 실 광고는 반드시 릴리스 빌드(--export-release)로만 나오며, 개발 중 디버그 빌드에선 계속 테스트 광고가 떠 안전합니다.

③ 릴리스 APK/AAB 빌드

cd godot
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
export ANDROID_HOME="$HOME/Library/Android/sdk"

# Play 업로드용 AAB (실 서명) — 별도 Android 프리셋 사용
godot --headless --path . --export-release "Android" \
  ../build/android/hitezero.aab
출시 빌드 격리 주의. AdMob 플러그인을 활성화한 이 작업 브랜치(neon-style-7cc31)에서 출시 AAB를 빌드하면 매니페스트에 INTERNET·AD_ID 권한과 AdMob SDK가 포함됩니다. 이는 의도된 동작(광고를 넣으니까)이지만, 기존 README/개인정보처리방침의 "완전 오프라인 / 데이터 수집 없음" 문구와 충돌합니다. 실제 출시 전 5장(데이터 보안 선언)과 문서를 반드시 업데이트하세요.

3사용자 동의 — UMP / GDPR / 테스트 기기

개인 맞춤 광고는 동의 수집이 필요합니다. 유럽(EEA)·영국 사용자에게는 필수이며, 미설정 시 광고가 노필(빈 배너)될 수 있습니다.

  1. AdMob → 개인정보 보호 및 메시지(Privacy & messaging). GDPR 동의 메시지(유럽 사용자 동의)를 생성. 개인정보처리방침 URL(예: https://hitezero.netlify.app/privacy)이 필요합니다.
  2. (미국) 주(州) 규정 메시지도 동일 메뉴에서 선택적으로 생성.
  3. UMP SDK 흐름 연동. 플러그인의 동의 API(ConsentRequestParameters, UserConsent)로 첫 실행 시 동의를 요청하도록 연결. (필요 시 별도 작업으로 도와드릴 수 있습니다.)
  4. 실 광고 검증 시 테스트 기기 등록. AdMob → 설정 → 테스트 기기에 내 기기 광고 ID 추가. 또는 코드에서 테스트 기기 목록 지정. 이러면 실 광고 단위라도 "Test Ad"가 떠 무효 트래픽 없이 검증 가능.
실 광고 단위를 본인이 직접 클릭하지 마세요. 무효 활동으로 간주되어 계정이 정지될 수 있습니다. 검증은 반드시 테스트 기기 등록 상태에서.

4Google Play Console 출시

play.google.com/console 에서 진행. 이미 출시 에셋(아이콘·스크린샷·키아트)은 store_assets/ 에 준비돼 있습니다.

  1. 개발자 계정 등록. 최초 1회 $25 등록비. (개인/사업자 신원 확인 필요할 수 있음.)
  2. 앱 만들기. 앱 이름 HiteZero, 언어, 무료 앱 선택.
  3. App Signing 활성화. Play App Signing 사용(권장). 업로드 키 인증서(SHA-256은 CI 문서에 기록됨)로 서명한 AAB 업로드.
  4. 스토어 등록정보 작성. 짧은/긴 설명, 스크린샷(store_assets/play_screen_*.png), 기능 그래픽(play_feature_1024x500.png), 아이콘(play_icon_512.png) 업로드.
  5. 콘텐츠 등급 설문 작성 → 등급 발급.
  6. 타겟 연령·광고 포함 여부 선언 → 5장 참고(광고 있음 = 예).
  7. 데이터 보안(Data safety) 양식 작성 → 5장 참고(AdMob은 데이터 수집함).
  8. 테스트 트랙부터. 내부 테스트(Internal testing) 트랙에 AAB 업로드 → 본인·소수 테스터로 실 광고/결제 확인 후 프로덕션 승격.

5데이터 보안 · 광고 선언 중요

광고 SDK를 넣는 순간, 앱은 더 이상 "데이터 미수집"이 아닙니다. Play 정책상 정확히 신고하지 않으면 앱 비공개/거부될 수 있습니다.

기존 문서 정합성. 현재 README.md·store_assets/privacy.html은 "광고 없음 / 완전 오프라인 / 데이터 미수집"을 명시합니다. 광고 도입 시 이 문구들을 모두 수정하고, 개인정보처리방침에 Google AdMob을 데이터 처리자로 명시해야 합니다. (AdMob 약관 + Google의 권장 문구 참고.)

6AdSense 설정 (웹 빌드 전용)

AdSense는 APK에 들어가지 않습니다. 브라우저 JavaScript 기반이라 모바일 앱(native)에선 동작하지 않습니다. 웹 버전(dist/netlify-polished 등)의 HTML 페이지에서 canvas 아래에 배치하는 별도 작업입니다.

  1. AdSense 가입·사이트 등록. adsense.google.com → 사이트 URL(예: hitezero.netlify.app) 추가.
  2. 사이트 승인 대기. AdSense는 콘텐츠 심사·승인이 필요합니다(수일~수주). 승인 전에는 광고가 빈 칸으로 나옵니다. 로컬/미승인 도메인에선 실광고 미표시.
  3. ads.txt 게시. 웹 루트에 ads.txt 추가 (7장 참고).
  4. 광고 단위 코드 삽입. AdSense에서 디스플레이 광고 단위 생성 → 스니펫을 게임 페이지의 canvas 아래에 삽입:
<!-- <head> 안 1회 -->
<script async src="https://pagead2.googlesyndication.com/pagead/js/adsbygoogle.js?client=ca-pub-XXXXXXXXXXXXXXXX"
        crossorigin="anonymous"></script>

<!-- 게임 canvas 아래(하단 배너 자리) -->
<ins class="adsbygoogle"
     style="display:block" data-ad-client="ca-pub-XXXXXXXXXXXXXXXX"
     data-ad-slot="ZZZZZZZZZZ" data-ad-format="auto" data-full-width-responsive="true"></ins>
<script>(adsbygoogle = window.adsbygoogle || []).push({});</script>
웹 빌드의 커스텀 HTML 셸에 이 슬롯을 넣는 작업은 원하시면 코드로 붙여드릴 수 있습니다(라이브 dist는 임의로 건드리지 않았습니다).

7app-ads.txt / ads.txt — 수익 보호 필수

광고 인벤토리 위조를 막는 인증 파일입니다. 미게시 시 수익이 크게 줄 수 있어 반드시 게시하세요.

모바일 앱(AdMob) → app-ads.txt

  1. Play 스토어 등록정보의 개발자 웹사이트에 도메인을 지정 (예: hitezero.netlify.app).
  2. 그 도메인 루트에 app-ads.txt 게시. AdMob → app-ads.txt 메뉴에서 본인 줄을 복사:
google.com, pub-XXXXXXXXXXXXXXXX, DIRECT, f08c47fec0942fa0

접속 확인: https://hitezero.netlify.app/app-ads.txt

웹(AdSense) → ads.txt

같은 한 줄을 파일명 ads.txt로 웹 루트에 게시 (Netlify면 dist/.../ads.txt 또는 publish 디렉터리에 포함).

8최종 체크리스트

  • AdMob 계정 생성 + 결제 정보 등록
  • AdMob에 Android 앱 추가 → App ID 확보
  • 배너 광고 단위 ID 생성·확보 (+선택 리워드)
  • android_export.cfg: is_real=true + [Release] 실 App ID
  • banner_ads.gd: android_real_banner_id 설정
  • GDPR/UMP 동의 메시지 생성 + 개인정보처리방침 URL 연결
  • 테스트 기기 등록 후 릴리스 빌드로 실 광고 1회 검증
  • README·privacy 문구를 "광고 있음/데이터 수집"으로 수정
  • Play 개발자 계정($25) + 앱 생성 + 등록정보·스크린샷
  • Play 데이터 보안 양식 정확히 작성 (광고/AD_ID = 예)
  • app-ads.txt(앱) / ads.txt(웹) 루트 게시
  • 내부 테스트 트랙 → 검증 → 프로덕션 승격
  • (웹) AdSense 사이트 승인 + canvas 하단 슬롯 삽입

9절대 하면 안 되는 것

실 광고를 본인이 클릭/반복 노출 — 무효 트래픽 → 계정 영구 정지 가능. 검증은 테스트 기기로만.
광고 도입 후 "데이터 미수집"으로 Play 신고 — 정책 위반. AdMob은 광고 ID를 수집합니다.
이 ad-test 브랜치를 그대로 release로 머지 — 출시 프로파일이 오염됩니다. 광고 포함 출시는 문서·선언을 갱신한 별도 정식 절차로.
app-ads.txt 누락 — 위조 인벤토리로 분류돼 수익 급감. 도메인 루트 게시 필수.
